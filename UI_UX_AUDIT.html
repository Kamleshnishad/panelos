PanelOS — UI/UX Audit & Enhancement Roadmap

Complete review of the Vue 3 ERP/SaaS front-end · grounded in the live screens + best-practice rules (UI/UX Pro Max)
Verdict: The product is functionally complete and structurally sound — consistent shell, sensible information architecture, and a clear brand. What separates it from a "looks like a polished enterprise SaaS" feel is mostly consistency + finish: emoji icons, duplicated button/card CSS across 29 files, two competing blues, plain "Loading…" text, and missing inline form errors. Fixing the P0 quick wins below will lift the perceived quality the most for the least effort.

Tier 0 — Quick wins (highest impact / least effort)

EnhancementWhat & whyWhereEffort
P0 Centralize design tokens + base components .btn { is redefined in 29 components; cards, status-chips and modals are re-styled per file. Move .btn / .btn-primary / .card / .chip / .modal / .input into styles.css as global classes driven by CSS variables. One source of truth → instant consistency + faster future work. rule: consistency, color-semantic styles.css + remove dup CSS M
P0 One brand blue Two primaries are in use: #1a237e (app, 14×) and #2B50E0 (PDFs/older components, 5×) plus #3d5af1. Pick one indigo, expose as --brand/--brand-600 tokens, replace all raw hexes. Inconsistent blue is the most visible "unpolished" tell. rule: color-semantic all components, PDF blades S
P0 Replace emoji icons with the SVG set Buttons use 📄 ⬇ 🛒 ⚠ 📦 ↻ ✕ 📤 💬 🚚 etc. Emojis render differently per OS and can't be themed. You already have a clean SVG ic map in AppShell.vue — extend it into a shared icons.js and use everywhere (refresh, close, download, pdf, send, truck…). rule: no-emoji-icons, icon-style-consistent ~all components M
P0 Skeleton loaders 20 components show a bare "Loading…". Replace with shimmer skeletons that mirror the layout (KPI cards, table rows). Removes the blank-flash and reads as premium. Add a reusable <SkeletonRow/> / .skeleton shimmer class. rule: progressive-loading, loading-states, content-jumping list + dashboard + detail pages M
P0 Inline field-level form errors Backend now returns the specific message, but forms still show ONE banner. Show the error under the offending field (red text + red border), and auto-focus the first invalid field on submit. Biggest day-to-day friction reducer. rule: error-placement, focus-management, aria-live-errors CompanySettings, all create/edit modals M

Tier 1 — High impact

EnhancementWhat & whyWhereEffort
P1 Data-table upgrade List tables (quotations/orders/invoices/customers) need: sortable headers (with aria-sort), a sticky header on scroll, right-aligned money columns with font-variant-numeric: tabular-nums, zebra rows, and a clear row-hover. Tables are the heart of an ERP — this is where polish shows. rule: sortable-table, number-tabular, data-table *List.vue, tables in managers M-L
P1 Richer empty states Replace italic "No records" with an icon + one-line explanation + a primary CTA ("No quotations yet — Create your first quote"). Turns dead-ends into next actions. rule: empty-states, empty-data-state all lists, dashboard widgets S
P1 Accessibility pass Add aria-label to every icon-only button (↻, ✕, ⬇ — currently ~20 labels for many more buttons); ensure a visible :focus-visible ring (2px brand) on all interactive elements; verify gray hint text meets 4.5:1; add role="alert"/aria-live to toasts & error banners. rule: aria-labels, focus-states, color-contrast, toast-accessibility global + AppShell + ui.js M
P1 Mobile-responsive tables Wide tables overflow on phones. Wrap in a horizontal-scroll container with a fade hint, OR switch to a stacked card layout below 640px (label:value rows). The shell already collapses the sidebar — finish the job on data views. rule: horizontal-scroll, responsive-chart, content-priority *List.vue, reports M
P1 Consistent toasts & confirms You have ui.js toasts/confirm — standardize: success/error/info variants with icon, auto-dismiss 4s, slide+fade in, stack top-right, aria-live="polite". Use confirm dialog for every destructive action (some still use native prompt — e.g. impersonate/cancel reasons). rule: toast-dismiss, confirmation-dialogs, success-feedback ui.js, SuperAdmin, InvoiceDetail S-M

Tier 2 — Polish & micro-interactions

EnhancementWhat & whyWhereEffort
P2 Motion system Define global tokens (--t-fast:150ms, --ease-out). Modals scale+fade from center, drawers slide, rows stagger-in 30ms, buttons press-scale 0.97. Wrap in @media (prefers-reduced-motion: reduce) to disable. Subtle motion = expensive feel. rule: duration-timing, modal-motion, reduced-motion, scale-feedback styles.css, modals M
P2 Typography scale + font Adopt a fixed type scale (12/13/14/16/20/24/30) and one weight ladder (400 body / 600 labels / 800 headings). Optionally load Inter (via font-display:swap) for a sharper SaaS look than system Segoe UI. Use tabular numerals for all money/quantities. rule: font-scale, weight-hierarchy, number-tabular styles.css S-M
P2 Spacing rhythm (4/8px) Page paddings vary (24/32/48 ad-hoc). Adopt a 4/8 scale via vars (--sp-2..--sp-8) for card padding, gaps and section spacing. Consistent rhythm reads as "designed". rule: spacing-scale, whitespace-balance global M
P2 Chart finish CSS bar charts need: tooltips on every bar (some have title), axis/value labels, an accessible text summary (aria-label), and a proper empty/loading state instead of a blank frame. Consider a tiny lib (e.g. uPlot/Chart.js) only if charts grow. rule: tooltip-on-interact, empty-data-state, screen-reader-summary, gridline-subtle BusinessReports, SuperAdmin M
P2 Detail-page hierarchy Invoice/Order/Quotation detail pages have many equal-weight cards. Introduce a sticky summary header (status + total + primary action), then secondary cards. One clear primary CTA per screen. rule: primary-action, visual-hierarchy InvoiceDetail, QuotationDetail M
P2 Global search / command palette The topbar "Jump to a section" can become a real Ctrl/⌘K palette: jump to any module + recent records. Power-user signal for an ERP. rule: search-accessible AppShell M-L

Suggested order

Sprint 1 (biggest visual lift): Tier 0 entirely — tokens + one blue + SVG icons + skeletons + inline form errors. Do the token/base-component centralization FIRST; every later change rides on it.
Sprint 2: Tier 1 — tables, empty states, a11y pass, responsive tables, toasts/confirms.
Sprint 3+: Tier 2 polish (motion, type, spacing, charts, detail hierarchy, ⌘K) then Tier 3 (dark mode, per-tenant theming).

Notes grounded in the codebase

  • .btn { appears in 29 component files — the single biggest consistency debt; fixing it is leverage for everything else.
  • Brand hex audit: #1a237e ×14, #2B50E0 ×5, #3d5af1 ×2 → unify to one token scale.
  • Plain "Loading…" in 20 components → skeletons.
  • ~20 aria-labels total against far more icon-only buttons → a11y gap.
  • Good foundations already present: CSS variables in styles.css, an SVG icon map in AppShell, a ui.js toast/confirm system, responsive sidebar, consistent page gutters — build the system on these.
