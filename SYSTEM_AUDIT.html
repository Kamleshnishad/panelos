PanelOS — System Audit

Daily-usage reliability · Huge-data scale · Concurrency · Security  |  Read-only, code-grounded (file:line) · Prepared for review — no changes made
Headline: The architecture is genuinely sound — real multi-tenant global scope (no cross-tenant IDOR found), no mass-assignment/SQL-injection holes, secrets not committed, most lists paginated, core tables indexed, transactions used widely, and a health-gated rolling deploy. The risks are not structural — they cluster in four fixable areas: (1) billing integrity & auth hardening, (2) concurrency locking (stock oversell, double-credit, double-conversion), (3) the dashboard/reports/exports loading whole tables with N+1 sums and no cache, and (4) operational safety that is wired-but-switched-off — most critically, the production scheduler never runs, so the "nightly backup" does not exist.

1 · Security — highest stakes; system onboards strangers via public trial signup

Critical SEC-C1 Billing verify trusts client-supplied plan / months / amount
Evidence: BillingController.php:106-137 verifies only the Razorpay signature (RazorpayClient.php:78-82) then activates the client-sent plan, months, amount — never re-fetches the order to confirm captured amount/plan.
Exploit: Checkout Starter×1 (cheapest), pay it, then call verify with the valid signature but plan=enterprise, months=36. Signature still matches (same order/payment id) → tenant activated on Enterprise for 3 years for the Starter price; revenue record corrupted.
Fix: Re-fetch the order via Razorpay API; derive plan/months/amount from server-side order notes + captured amount (checkout already stamps these at BillingController.php:86). Never trust client plan/amount on verify.
High SEC-H1 2FA OTP written to application log on every login (production)
Evidence: AuthController.php:153 logs "2FA code for {email}: {code}" unconditionally; prod uses MAIL_MAILER=log so the email body with the code also lands in logs.
Impact: Anyone with log access (ops, aggregator, a future log-exposure bug) fully bypasses 2FA.
Fix: Remove the unconditional log; gate any dev fallback behind app()->environment('local'); use a real mailer in prod.
High SEC-H2 OTP has no per-account attempt lockout (brute-forceable)
Evidence: AuthController.php:157-170; route throttle is only throttle:10,1 per IP (routes/api.php:44). 6-digit code, 10-min window, no failure counter.
Fix: Add an attempt counter; invalidate the OTP after ~5 fails; add a per-email throttle (login already has one at routes/api.php:32-38).
High SEC-H3 Production served over plain HTTP — no TLS / HSTS / secure cookies
Evidence: APP_URL=http://208.87.135.37:3030, SESSION_DRIVER=cookie, no forceScheme('https') anywhere.
Impact: Sanctum tokens + 30-min impersonation tokens travel in cleartext; cookies not marked secure.
Fix: Terminate TLS (domain + cert), set APP_URL https, force https scheme, enable SESSION_SECURE_COOKIE.

2 · Concurrency — correctness under simultaneous users; lockForUpdate appears exactly once in the whole codebase

High CONC-H1 Stock deduction is read-modify-write with no row lock → oversell / lost-update
Evidence: StockService.php:37-61 / 114-138 / 183-204 — find() → check qty → update(qty ± n) inside a transaction but without lockForUpdate. Production consumption routes through these (ProductionMaterialService.php:99-123).
Race: Coil = 100kg. Run A & Run B start together, both read 100, both pass availability, A writes 20, B (stale 100) also writes 20 → 160kg consumed, stock shows 20 not −60. Valuation & BOM now wrong.
Fix: ...->lockForUpdate()->findOrFail($id) inside the existing transaction, or atomic UPDATE … SET qty=qty-? WHERE qty>=? and check affected rows.
High CONC-H2 Razorpay double-credit — verify + webhook, no idempotency on payment id
Evidence: Two entry points both call activate(): BillingController.php:106-137 (client) and :140-164 (webhook). activate() stacks months onto current end date (SubscriptionService.php:19-31); webhook's only guard is status !== 'active' — which verify already cleared. razorpay_payment_id stored but never checked for reuse.
Impact: Webhook racing/after verify → customer credited twice (double months) + a duplicate subscription invoice number.
Fix: Dedupe on razorpay_payment_id (unique index on subscription_payments.reference or explicit check inside a transaction) before activating, in both paths. (Stripe path already does this correctly — PaymentGatewayService.php:171-228.)
High CONC-H3 Subscription invoice number race with no unique backstop
Evidence: SubscriptionService.php:60-69 reads max then +1; subscription_payments.invoice_no has no unique constraint (2026_06_15_000004…:22). Whole activate() is not in a transaction (:19-57).
Impact: Two near-simultaneous activations mint the same PINV-2026-NNNN with zero DB protection — silent duplicate GST invoice numbers (compliance issue).
Fix: Add unique index on subscription_payments.invoice_no; wrap activate() in DB::transaction with a locked counter.
High CONC-H4 Double quotation→order & double-dispatch via pre-transaction exists() check
Evidence: OrderService.php:18 checks $quotation->orders()->exists() before the transaction; no unique on orders.quotation_id. Same shape for dispatch: DispatchService.php:42 checks batch_id exists but only dispatch_no is unique.
Impact: Two concurrent conversions → two orders (later two invoices) for one quotation; one batch dispatched twice.
Fix: Unique index on orders.quotation_id and on dispatches.batch_id; load the parent with lockForUpdate() before the check.

CONC-M1MediumDoc numbering (invoice/quote/order/PO/run/batch) = read-max-then-+1 with no lock; unique indexes block duplicates but the loser gets an unhandled 500 (and for invoices the whole creation rolls back) under burst load.Locked counter row, or catch unique-violation & retry the number.
CONC-M2MediumCredit-limit TOCTOU — QuotationController.php:170-198 checks outstanding then creates order; two concurrent orders both pass → limit breached.Re-check credit inside the order transaction after locking the customer's outstanding scope.
CONC-M3MediumSubscriptionService::activate() and BillingController::verify() coupon-increment+activate are not atomic → company marked active with no payment record on partial failure.Wrap both in DB::transaction.

3 · Huge data / Scale — behaviour at hundreds of thousands of rows after years of use

Critical SCALE-C1/C2 Dashboard loads all invoices + per-invoice SUM N+1 — twice — with no cache
Evidence: ReportingService.php:148-156 loads every non-draft invoice (no date floor, 4-level eager load) then runs PaymentTransaction::where('invoice_id',$inv->id)->sum() per invoice; DashboardService.php:219-231 does a second full scan + same N+1; zero Cache::remember (DashboardController.php:16-24).
Impact: At 200k invoices = one huge hydration + ~400k queries on every dashboard load/poll, by every user.
Fix: One grouped SQL aggregate for paid-per-invoice (or a denormalized balance_due column); compute aging in SQL; cache dashboard 60s per company; delete the duplicate method.
Critical SCALE-C3 CSV & Tally exports buffer the entire table in memory
Evidence: ExportController.php:39-74 ->get()->map() + the csv() helper concatenates everything into one string; customers() has no date filter; invoices() eager-loads 4 deep + sums per row. Tally builds one giant XML/CSV string (ReportingService.php:629-672).
Impact: Three in-memory copies of a full table → OOM / timeout on large tenants; a fiscal-year Tally export OOMs.
Fix: StreamedResponse + fputcsv driven by lazy()/chunkById(); queue very large exports.

4 · Daily-operations reliability — all-day factory use; safety net is wired but switched off

Critical OPS-C1 The production scheduler never runs → backups & reminders never fire
Evidence: Console/Kernel.php:10-38 schedules 4 jobs (incl. nightly DB backup 01:30), but nothing runs schedule:run — the container (supervisord.conf) runs only php-fpm + nginx. No cron, no schedule:work.
Impact: The advertised nightly DB backup does not exist in prod; payment + SMS reminders never send. Silent total failure of every recurring task.
Fix: Add a supervisord program running the scheduler each minute (or real cron / schedule:work); verify a backup file appears.
Critical OPS-C2 Backups (even if run) are same-host, no off-site, restore never tested
Evidence: BackupDatabase.php:27 writes to storage/app/backups — same host/volume as the DB. Deploy has no pre-migrate dump; no restore procedure anywhere.
Impact: A disk failure / bad migration / git reset --hard mishap loses all factory data with no recovery. Same-host copy is not a backup.
Fix: Ship dumps off-host (S3/rsync); take a pre-migrate --force dump in deploy; document & test a restore.

Fix-risk filter — how much each fix can disturb the running system

How to read this: severity = how bad the problem is; fix-risk = how likely the fix is to break something that currently works. 🟢 Easy/Safe = additive or mechanical, no behavior change — ship freely. 🟡 Moderate = focused change, small blast radius, needs testing. 🔴 Hard/Risky = touches a core/live flow (money, inventory, infra, auth) — real chance of regression, test hard + have rollback.
Tip: Easy ≠ unimportant — several Easy fixes are also Critical/High. Bank all the 🟢 first (safe wins), then schedule 🟡, treat 🔴 as projects.
